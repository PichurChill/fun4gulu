# EHub App Front Agent Guide

## 项目定位

## 第一原则

- 先回到原始需求、业务目标和用户动机，不要直接套模板实现。
- 如果需求本身不完整、目标不明确、或者实现路径会改变业务语义，先停下来澄清。
- 坚持 fail-fast，尽早暴露问题，不要用兜底、兼容层、补丁式逻辑掩盖真实问题。

## 方案约束

- 不做兼容性方案，不做补丁式修复，不做降级兜底。
- 不过度设计，始终选择满足需求的最短实现路径。
- 不擅自扩展需求边界，不引入用户未要求的能力、流程或抽象。
- 任何方案都必须做全链路逻辑校验，确认输入、状态流转、界面行为和输出结果一致。

## 仓库工作方式

- 非用户明确要求时，不修改构建链路、依赖版本、脚手架配置和跨端平台配置。

## Agent 目录约定

- `AGENTS.md` 是唯一主脑，所有全局协作规则、项目约束、决策原则都维护在这里。
- `CLAUDE.md` 只是指向 `AGENTS.md` 的软链，不维护独立内容。
- `.agents/` 是项目内自定义 skill 的唯一真源目录。
- `.claude/` 只作为 Claude 适配层，放 Claude 私有配置和指向 `.agents/` 的软链，不放 skill 真文件。
- 不允许同时维护 `skills/` 和 `.agents/` 两套 skill 真源。

## 技术上下文

## 修改要求

- 先理解现有实现和真实问题，再动代码。
- 尽量做局部、直接、可验证的改动，避免“顺手重构”。
- 涉及业务逻辑时，必须说明触发条件、状态变化和最终结果是否闭环。
- 若发现现有代码与需求冲突，以需求真实意图为准，但必须明确指出冲突点。

## 输出要求

- 先给结论，再给关键依据。
- 说明本次改动影响了哪个子应用、哪些核心文件、解决了什么问题。
- 如果做了验证，明确写出验证方式；如果没验证，明确说明缺口。

## 禁止事项

- 禁止为了“保险”加入未被要求的 fallback、default 分支保护或兼容旧逻辑。
- 禁止在未确认前提下修改多个业务端。
- 禁止把不确定的猜测写进代码。
- 禁止在未充分理解调用链前直接重构公共逻辑。
